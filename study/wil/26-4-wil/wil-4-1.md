# WIL: 이번주 공부(4월 1주차)

## WIL: 이번주 공부 (4월 1주차)

TL;WR 권한 관리 도메인 설계에서 3가지 병합 패턴을 비교 분석하고 확장 가능한 구조로 정했으며, 모니터링 대시보드의 숨겨진 오류(평균을 P95로 표기)를 발견하고 G1GC 영역별 분석 패널을 추가했다. 12개 도메인 컨텍스트를 병렬로 생성하면서 체계적인 문서화가 코드 품질 검증 도구임을 체득했다.

### 좋았던 점 & 배웠던 점

#### 1. 권한 병합 패턴의 선택이 미치는 설계 영향

개인 권한과 그룹 권한을 합치는 방식에서 3가지 패턴(OR 합집합, 개인 오버라이드, 택 1)을 분석했다. 처음엔 "무엇이 가장 융통성 있을까" 중심으로 생각했지만, AWS IAM과 Azure RBAC라는 업계 표준이 모두 패턴 A(OR 합집합)를 선택한 이유를 학습했다.

가장 인상적이었던 부분은, 확장 가능성을 염두에 두고 기존 로직(`PermissionDomainService`)은 한 줄도 건드리지 않고 새로운 체크 레이어(`UserPermissionDomainService`)를 추가하는 구조다. 이렇게 하면 나중에 더 복잡한 정책(예: 개인이 그룹 권한을 제한)으로 확장할 때도 병합 로직만 교체하면 되는 유연성이 생긴다.

#### 2. 기획 공백을 기술 선택지로 채우는 방법론

권한 관리 주체(누가 개인 권한을 설정할 수 있는가)가 기획팀에서 정해지지 않았을 때, 세 가지 방안(중앙 집중, 위임형, 혼합)의 트레이드오프를 문서화한 후 기획에게 넘기는 것이 가장 효율적임을 느꼈다.

그리고 API 구조와 Controller 레이어 검증 조건을 분리해서, "지금 확정해야 할 부분"과 "나중에 교체 가능한 부분"을 명확히 구분하는 것의 중요성을 이해했다. 기획 결정을 기다리면서도 개발을 진행할 수 있는 탄력성이 생긴다.

#### 3. /new-context를 통한 문서화 프로세스의 실질적 가치

새로운 도메인(individual-permission)을 추가할 때, 검증 질문 4개(문제, 현재 프로세스, 미실행 영향, 규모)를 통해 요구사항을 명확히 하고, 그 결과를 README, spec, architecture, glossary로 구분된 문서로 체계화했다.

인상적이었던 점은 유즈케이스 8개와 시퀀스 다이어그램 5개를 그리는 과정에서 누락된 요구사항 3개(사용자/메뉴 삭제 시 개인 권한 처리)가 자연스럽게 드러났다는 것이다. 코드부터 짜면 나중에 발견될 오류가, 문서화 단계에서 먼저 캐치된다.

#### 4. Grafana 메트릭 오류의 숨은 위험성

hikaricp-deep-dive 대시보드의 "테넌트별 응답 시간 (p95)" 패널이 `_sum/_count`로 평균을 계산하고 있었다. 이는 P95 계산식(`histogram_quantile(0.95, rate(_bucket))`)과는 근본적으로 다르며, 트래픽 스파이크 시 평균과 P95는 10배 이상 차이가 난다.

운영 팀은 이 대시보드를 보고 SLA를 판단하고 있었을 텐데, 표기는 P95인데 실제로는 평균을 보고 있었던 것이다. 이런 오류는 업무 판단에 직접 영향을 미친다는 점에서 "대시보드는 코드와 같은 수준의 검증이 필요하다"는 깨달음을 얻었다.

#### 5. G1GC 메모리 영역별 분석의 구체화

D2 블로그의 jstat 기반 GC 모니터링을 현재 Prometheus 메트릭과 매핑했다. G1GC는 Serial GC와 달리 S0/S1을 물리적으로 구분하지 않으며, Young GC(`end of minor GC`)와 Full GC(`end of major GC`)를 action 라벨로 구분할 수 있다는 점을 체계적으로 정리했다.

특히 Full GC의 누적 횟수가 0이 아니면 메모리 누수를 의심할 근거가 된다는 통찰, Old Gen 사용률 지속 상승 = 메모리 누수 신호라는 실전 판단 기준이 얻어졌다.

#### 6. 대규모 도메인 문서화 시 병렬 처리의 효율성

12개 도메인을 순차적으로만 처리하면 시간이 오래 걸리지만, 에이전트 병렬화를 활용하여 (Backend 분석, Frontend 분석을 동시 수행 + 독립적 도메인 병렬 처리로) 작업 시간을 줄일 수 있음을 실감했다.

또한 재고와 입고처럼 의존성이 있는 도메인의 경우 순서를 맞춰서 처리해야 한다는 학습, 병렬 처리를 위해서는 의존성 매핑이 먼저 필요하다는 원칙도 확립했다.

#### 7. 문서화가 코드 품질 검증 도구가 된다는 경험

12개 도메인을 문서화하면서 기존 코드의 불일치를 자연스럽게 발견했다:

* 검사 도메인: 자주검사만 List를 String으로 직렬화하는 불일치
* 생산실적: 모달 수 불일치 (문서: 8개 → 실제: 11개)
* 유지보수: 패키지명 오타(mainteance), API 누락(DELETE 엔드포인트)

체계적인 컨텍스트 문서 작성 과정이 단순 "기록"이 아니라 **품질 검증의 자동화 도구**로 기능함을 체득했다.

#### 8. 도메인 간 반복되는 설계 패턴의 인식

12개 도메인을 분석하면서 다음과 같은 패턴들을 발견했다:

* **워터폴 배분**: 판매, 출하 도메인에서 반복
* **2그룹 패턴**: 검사 도메인 (생산 기반 vs 수불 기반)
* **상태 계산의 복잡도**: 구매 도메인 (12개 외부 서비스 의존)

같은 패턴이 여러 곳에서 나타나면, 공통 컴포넌트로 추상화할 기회를 발견하는 계기가 된다.

### 아쉬운 점

#### 1. 권한 관리 정책의 확정 없이 기술 결정을 내렸다

패턴 A(OR 합집합)를 선택했지만, 실제로 운영에서 "개인 권한이 제한 역할을 해야 한다"는 요청이 나올 가능성이 있다. 기획과 더 깊이 있는 대화 없이 "일단 A로 가고 나중에 바꾼다"는 결정은 기술 부채가 될 수 있었다.

#### 2. 대시보드 오류 발견이 너무 늦었다

메트릭 계산식 오류(평균을 P95로 표기)를 이번에 발견했지만, 실제로는 대시보드가 생성된 이후로 운영팀이 이를 기반으로 판단해오고 있었을 것이다. 대시보드 검증 프로세스가 부재했던 것이 아쉽다.

#### 3. Logbook 기반 URI 추출 불가능 문제의 미해결

app-log-explorer의 "요청 URI Top 10" 패널이 Logbook의 logger 레이블 구조 때문에 데이터를 뽑을 수 없다. HttpLogFormatter 커스텀이 필요하지만 현재 보류 상태다.

#### 4. 도메인 컨텍스트 기존 파일의 세밀한 검증 없음

12개 도메인 신규 생성 후 기존 7개 도메인을 일부 수정했지만, 22개 전체에 대한 완전한 검증을 하지 못했다. 특히 개발/운영 환경 간 라벨 체계 차이처럼 후발견 오류가 있을 수 있다.

#### 5. 도메인 간 의존성 관계를 명시적으로 문서화하지 않음

구매→주문, 재고→입고 같은 의존성은 인식했지만, 이를 컨텍스트 문서의 별도 섹션으로 명확히 하지 않았다. 나중에 새로운 도메인을 추가할 때 의존성 매핑 작업이 다시 필요할 것 같다.

***

### 종합 정리

이번 주는 **"설계 결정의 트레이드오프를 명확히 하는 것"과 "문서화를 통한 품질 검증"이 핵심**이었다.

권한 관리 도메인에서는 "가장 좋은 선택"이 아니라 "현재 최선의 선택이면서 나중에 확장 가능한 구조"를 찾는 것이 얼마나 중요한지 느꼈다. 기획이 확정하지 않은 부분도, 아키텍처 수준에서 플러그 앤 플레이하게 만들 수 있다면 개발은 중단되지 않는다.

모니터링 대시보드의 평균과 P95 오류는 작은 것처럼 보이지만, 운영 판단에 직접 영향을 미친다. \*\*"대시보드는 단순 시각화가 아니라 비즈니스 결정의 근거"\*\*라는 책임감을 다시 확인했다.

12개 도메인 컨텍스트 생성은 가장 규모가 컸던 작업인데, 병렬 처리와 체계적인 검증 질문의 조합으로 효율성을 높일 수 있었다. 더 중요한 것은, 문서화 과정에서 기존 코드의 불일치가 자동으로 드러난다는 것이다. 이는 단순 "기록"이 아니라 **코드 품질을 검증하는 도구**로서의 문서의 가치를 보여준다.

앞으로의 개발에서 지킬 원칙들:

1. 기획 공백은 "기술 선택지의 문서화"로 채우기
2. 대시보드의 메트릭 계산식을 코드 리뷰처럼 검증하기
3. 새로운 도메인 추가 전에 전체 의존성 지도 그리기
4. 문서화를 품질 검증의 첫 단계로 삼기
5. 대규모 작업일수록 병렬화와 에이전트 활용 적극화
